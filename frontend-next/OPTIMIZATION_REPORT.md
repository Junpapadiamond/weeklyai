# WeeklyAI 前端优化分析报告

> 基于代码审查 + 线上站点 (weeklyai-six.vercel.app) 的完整审视
> 日期: 2026-03-09

---

## 一、总体印象

WeeklyAI 目前的代码质量不错——组件拆分清晰、CSS 变量体系完整、可访问性有基础投入。但**视觉呈现上存在「功能堆叠」问题**：每个模块都在努力展示很多信息，导致整体观感像一个内部 dashboard，而不是一个面向 PM 的灵感发现工具。

核心矛盾：**你想让用户「5 分钟看完全球 AI 黑马」，但首屏花了 80% 空间在统计数据和搜索栏上，真正的黑马产品被推到了 fold 以下。**

---

## 二、问题逐一拆解

### 问题 1：Hero 区占据整个首屏，核心内容被挤出视野

**现状（代码位置：`home-client.tsx` L471-528, `home.css` L1-178）**

Hero 区内部包含 **7 层信息**：
1. kicker 标签 "GLOBAL SIGNAL BOARD · WEEKLYAI"
2. 大标题 "发现全球最新 AI 产品"
3. 副标题说明
4. 数据更新时间
5. 4 个统计卡片（本周新增 / 获得融资 / 覆盖地区 / 累计收录）
6. "本周偏热" 热词
7. ChatBar 搜索组件 + 4 个 suggested prompt 按钮

加上 p5.js 粒子动画背景（`hero-canvas.tsx` 230 行），hero `min-height: 360px`，实际渲染高度约 **450-500px**。在 1080p 屏幕上，首屏完全被 hero 占满，用户看不到任何一个产品。

**问题本质**：Hero 试图同时充当「品牌宣传」「数据仪表盘」「搜索入口」「趋势预览」四个角色。

**建议方案**：

A. **砍掉 Hero 中的统计卡片区**。"本周新增 0 款"这个数字对用户无意义（而且经常是 0）；"累计收录 542 款"是虚荣指标。如果真要保留，可以作为 footer 底部的一行小字。

B. **把 ChatBar 移到 header 或 dark horse section 上方**，而不是嵌在 hero 中。搜索是工具性动作，不需要占据视觉中心。

C. **Hero 压缩为 2 行**：大标题 + 一句话副标题，去掉 kicker、去掉热词、去掉统计。高度从 ~450px 缩减到 ~160px。

D. **考虑完全移除 p5.js 粒子动画**。暗色主题下几乎不可见（`opacity: 0.32`），但加载了 p5 库（~800KB gzipped ~270KB）+ 230 行运行时代码 + RAF 循环。投入产出比极低。

**改后首屏预期**：用户打开即可看到 Hero 一行 + 2-3 个黑马产品卡片。

### 问题 2：黑马卡片信息过载

**现状（代码位置：`DarkHorseSpotlightCard` L91-217, `home.css` L186-396）**

每张卡片包含 **11 个信息单元**：
1. 排名 badge (#1, #2...)
2. 国旗（右上角）
3. Logo（64px）
4. 产品名
5. 分类标签 (e.g. "硬件")
6. 一行 description（line-clamp 1）
7. region 标签 ("United States")
8. funding 标签 ("$500M Series E")
9. 评分 badge ("5分")
10. why_matters 段落（line-clamp 3 + 展开按钮）
11. 详情 / 官网 两个按钮

在一个 ~300px 高的卡片里塞了 11 个元素。用户的眼睛不知道先看哪里。

对比参考：Product Hunt 的卡片只有 **4 个元素**（logo + 名字 + 一句话描述 + upvote）。

**建议方案**：

A. **合并 region + funding + score 为一行 metadata**，用文字而非 badge 呈现：`🇺🇸 $500M · 5分`，一行搞定。

B. **去掉 description 行**。当 `why_matters` 已经存在时，description 和 why_matters 内容高度重叠。只保留 why_matters 即可。

C. **rank badge 简化**：`#1` 不需要 pill 样式 + 蓝色背景。用序号数字即可，或直接去掉——卡片的位置已经隐含了排名。

D. **卡片最小高度去掉**：`min-height: 332px` 导致内容少的卡片有大片空白。改为内容自适应。

E. **Featured 卡片（Top 3）和普通卡片差异过大**：Featured 用 128px logo + 3 列展示，普通用 64px logo + 2 列。建议统一为一种卡片样式，通过排序位置体现重要性，而不是两套布局。

### 问题 3：「更多推荐」区块可能未渲染 / 底部断裂

**现状（代码位置：`HomeClient` L632-761）**

从线上截图看，滚过黑马卡片后，下方就是大片黑色空白。代码里有 `trending-section`（更多推荐）和 `section--linkout`（发现/新闻链接 banner），但可能因为数据问题没渲染出来。

**可能原因**：
- `allProducts` 为空或过滤后无结果
- Intersection Observer sentinel 未触发分页

**建议排查**：检查 `productPool` 的计算逻辑（L371-452 附近）。如果 `allProducts` 全部是 dark horse（4-5分），那 `tierFilter=all` 时 "更多推荐" 和黑马区内容完全重复。

**建议方案**：

A. 即使数据为空，也应该有一个 footer section（About / RSS link / Last updated / 社交链接），避免页面突然结束。

B. 考虑在黑马区下方直接展示"更多推荐"的前几个，不依赖 Intersection Observer 触发——即首次渲染就加载第一页。

### 问题 4：视觉系统碎片化

**tokens.css 分析**

设计系统的 token 定义本身不错，但实际使用中存在碎片化：

**圆角不一致**：
- 定义了 `--radius-sm: 12px`, `--radius-md: 16px`, `--radius-lg: 22px`
- 但代码中大量硬编码圆角：`border-radius: 18px`（picks-row rank）, `border-radius: 14px`（picks-row logo）, `border-radius: 24px`（featured logo）, `border-radius: 10px`（swipe card ghost logo）
- **建议**：用 token 变量统一，减少魔法数字

**颜色混合过度使用**：
CSS 中大量使用 `color-mix(in oklab, ...)`，例如：
```css
border-color: color-mix(in oklab, var(--primary) 34%, var(--border));
background: color-mix(in oklab, var(--surface-strong) 96%, transparent);
color: color-mix(in oklab, var(--primary) 74%, var(--text));
```
总共有 **60+ 处** color-mix 调用，每次比例不同（74%, 78%, 72%, 86%...）。这导致：
- 颜色无法形成记忆——用户分不清哪些颜色代表哪些含义
- 维护困难——改一个主色要检查 60+ 处
- **建议**：提炼出 5-6 个语义色（`--primary-subtle`, `--primary-muted`, `--success-subtle` 等）作为 token，替代散落的 color-mix

**间距规范缺失**：
- 没有定义间距 scale（如 4/8/12/16/24/32/48）
- 代码中间距全是 magic number：`0.46rem`, `0.28rem`, `0.68rem`, `0.86rem`...
- **建议**：定义 `--space-1: 0.25rem` 到 `--space-12: 3rem` 的间距 scale

### 问题 5：交互体验问题

**5a. 筛选器控件太多**

"更多推荐"区有 **4 层筛选**：
1. 层级筛选（全部 / 黑马 / 潜力股）
2. 排序（综合 / 热度 / 时间）
3. 一级分类（全部 / 软件 / 硬件）
4. 方向标签栏 + "更多方向" modal

加上黑马区的 3 个筛选按钮（全部 / 硬件 / 软件），整个页面有 **7 个独立的筛选控件**。对于一个"5 分钟浏览"的产品来说，这太多了。

**建议**：
- 黑马区和更多推荐区的"硬件/软件"筛选是重复的，合并为页面级 global filter
- "方向标签栏"只在更多推荐区有用，可以用 inline tag cloud 代替专门的 filter bar
- 排序 select 可以降级为卡片上的小图标菜单

**5b. 滚动节奏不佳**

Hero (450px) → 黑马区头部（标题+描述+筛选，约 120px）→ Featured 3 列大卡 → 2 列普通卡 → 链接 banner → 更多推荐头部（标题+描述+4层筛选，约 200px）→ 列表

用户需要滚过 ~700px 的"非内容区"才能从 hero 到第一个产品卡片，再滚过 ~200px 的控件区才能看到更多推荐的第一个产品。

**建议**：减少 section header 的描述性文字。`section-desc` 和 `section-micro-note` 可以合并或删除。"先看 Top 3 Featured，再浏览其余黑马" 这样的指导性文字对用户帮助不大。

**5c. 暗色模式默认问题**

站点默认深色主题。深色模式下，卡片的 `var(--surface): #1c2128` 与背景 `var(--bg): #0f1117` 对比度很低，卡片不够突出。hover 时的 `border-color` 变化（红色 tint）在暗色下不明显。

**建议**：
- 考虑默认浅色主题（更适合内容浏览型产品）
- 如果保留暗色默认，增加卡片与背景的亮度差（surface 提亮到 `#222933` 左右）

**5d. p5.js 动画性能**

`hero-canvas.tsx` 创建 520-1300 个粒子，每帧更新位置 + 连线计算，30 FPS。在移动端虽然不渲染（`isMobileHero` 判断），但在低端笔记本上仍会消耗 CPU。

如果决定保留，建议：
- 粒子数减半（260-650）
- 考虑用 CSS gradient animation 替代 p5.js（体积从 ~270KB 降到 0）

### 问题 6：CSS 架构问题

**6a. 重复定义**

`home.css` 中 `.hero` 被定义了两次（L5-18 和 L996-1004），属性有冲突：
- 第一次：`min-height: 330px`, `margin-top: 1.25rem`
- 第二次：`min-height: 360px`, `margin-top: 1.4rem`
- 后定义覆盖前定义，但这样容易造成混乱

`hero-layout`, `hero-chat-slot` 也有类似重复。

**建议**：合并为单一定义。

**6b. base.css 过大（61.8KB）**

`base.css` 包含了大量全局组件样式（按钮、badge、filter、navbar、favorites panel 等）。这应该按组件拆分为独立 CSS 文件。

**6c. home.css 1506 行**

单个 CSS 文件 1500+ 行，包含了首页 + discover 页 + swipe 卡片的所有样式。建议拆分为 `hero.css`, `darkhorse.css`, `swipe-card.css`, `picks.css`。

## 五、不建议改的地方

1. **整体配色方案**：red primary + dark theme 的组合有辨识度，不需要换。
2. **字体选择**：Plus Jakarta Sans + Noto Sans SC 的搭配合理。
3. **Swipe Card 组件**：`discovery-deck.tsx` 的物理引擎和手势处理写得很精细，不需要动。
4. **组件拆分架构**：Server/Client 分离、dynamic import 等模式正确。
5. **国际化方案**：`t()` 内联翻译虽然不是最优，但对现有规模够用。
